# 上下文工程规范

> 本文规范 Skill 如何装载、筛选、隔离和压缩上下文，避免把私有知识或大量无关材料直接塞进主 Agent。

## 1. 职责

上下文工程解决三个问题：

- Agent 需要哪些资料才能做好任务。
- 这些资料应该何时读取。
- 读取后应以什么形式进入主流程。

公开 Skill 不应硬编码作者私有知识。它应声明资料类型、读取方式和输出格式，让使用者提供自己的资料源。

## 2. 核心原则

### 2.1 入口不装长上下文

`SKILL.md` 只写“需要哪类上下文”和“何时读取”。不要把长参考资料塞进入口。

### 2.2 确定性规则可直读

格式红线、命名规则、输出模板、字段定义这类确定性规则，适合由主 Agent 直读原文并遵守。

### 2.3 创造性材料先隔离

竞品案例、历史作品、用户反馈、行业资料这类需要筛选和判断的材料，适合交给 SubAgent 提炼为执行指令。

### 2.4 输出是执行指令

上下文步骤的输出不是研究报告，而是给后续步骤用的操作指令。

错误输出：

```text
我阅读了 12 篇文章，发现它们都很强调问题意识。
```

正确输出：

```text
本次报告开头应直接写用户痛点，不写行业背景。标题使用“问题 + 解决结果”结构。不要使用泛化口号。
```

## 3. 四种装载方式

```text
直读
  适合确定性规则。
  例如格式规范、输出模板、字段定义。

固定路径
  适合用户或配置能提供明确文件路径的资料。
  例如 references/style.md、docs/policy.md。

检索发现
  适合需要按关键词从大量资料中找相关内容的场景。
  例如历史案例、知识库、网页资料。

混合模式
  先读固定分析框架，再检索案例，用框架分析案例。
  适合竞品分析、内容策略、审查规则扩展。
```

## 4. 资料维度

一个 Skill 可以声明这些维度，按需选择：

```text
identity
  身份、角色、品牌定位。

style
  语言、结构、格式、表达风格。

audience
  目标读者、用户画像、痛点。

rules
  平台规则、格式红线、合规要求。

examples
  正例、反例、期望输出。

history
  已有作品、已有记录、避免重复。

reference
  外部资料、公开文档、研究材料。

constraints
  预算、时间、工具、权限、平台限制。
```

公开 Skill 应让这些维度成为可配置输入，而不是写死作者自己的资料。

## 5. 上下文步骤

长上下文装载应作为独立步骤。

```text
workflow/
  step01-prepare.md
  step02-load-context.md
  step03-execute.md
```

`step02-load-context.md` 应输出：

```text
step02-load-context/
  context.md
  sources.json
  skipped-sources.json
```

`context.md` 的推荐结构：

```markdown
# 执行上下文

## 装载状态

- 已读取：
- 已跳过：
- 失败：

## 本次任务约束

## 必须遵守

## 推荐策略

## 禁止事项

## 后续步骤读取方式
```

## 6. sources.json

上下文来源应可追踪。

```json
{
  "sources": [
    {
      "id": "style",
      "path": "references/style.md",
      "mode": "direct-read",
      "status": "loaded"
    }
  ],
  "skipped": [
    {
      "id": "history",
      "reason": "用户未提供历史资料目录"
    }
  ]
}
```

## 7. 直读规则

适合直读：

- 字段定义。
- 输出模板。
- 平台格式规则。
- 安全红线。
- 用户明确提供的短文档。

不适合直读：

- 大量案例。
- 多篇长文。
- 需要综合判断的资料。
- 会污染主 Agent 判断的原始素材。

## 8. SubAgent 提炼规则

SubAgent 适合：

- 阅读大量资料。
- 压缩上下文。
- 提取与任务相关的部分。
- 生成下一步可执行指令。

SubAgent 输出必须短、具体、可执行。

推荐输出：

```text
本次执行指令
  1. 只使用公开资料，不引用内部路径。
  2. 报告先列阻塞问题，再列建议。
  3. 每个阻塞问题必须包含文件路径和证据。

禁止
  - 不要复述全部资料。
  - 不要输出无法验证的判断。
```

## 9. 上下文预算

默认原则：

- 入口文件少于 500 行。
- 单个步骤文件尽量少于 300 行。
- 长参考资料按需读取。
- SubAgent 输出尽量控制在 2000 字以内。
- 不为了“完整”牺牲执行清晰度。

预算不只是 token 问题，也是注意力问题。即使上下文窗口很大，低相关材料也会降低执行稳定性。

## 10. 配置方式

在 `config/default.json` 或运行时配置中声明上下文来源：

```json
{
  "context": {
    "style_path": "references/style.md",
    "examples_dir": "examples/",
    "history_dir": null,
    "max_sources": 10
  }
}
```

规则：

- 配置中只放路径、开关、限制和默认值。
- 不放真实凭据。
- 不放私有不可分发路径。
- 本次运行解析出的实际路径写入 `state/config.json`。

## 11. 公开发布边界

公开 Skill 可以写：

```text
如果用户提供 references/style.md，则读取并遵守。
如果没有提供，则使用本 Skill 内置默认模板。
```

不要写：

```text
读取作者本机某个知识库路径。
读取作者私人品牌文件。
读取客户历史案例。
```

## 12. 失败处理

上下文装载失败时：

1. 判断是否阻塞主任务。
2. 阻塞则停止，说明缺失资料。
3. 不阻塞则记录到 `skipped-sources.json`。
4. 后续报告中写明跳过的检查或资料。

不要静默忽略。

## 13. 检查清单

```text
结构
  [ ] 长上下文有独立步骤。
  [ ] context.md 输出为执行指令。
  [ ] sources.json 记录来源。

装载
  [ ] 确定性规则直读。
  [ ] 创造性材料隔离提炼。
  [ ] 大量资料不直接塞入主 Agent。

公开
  [ ] 不硬编码私有路径。
  [ ] 不硬编码客户数据。
  [ ] 不要求读者拥有作者知识库。

失败
  [ ] 缺资料时有停止或降级策略。
  [ ] 跳过资料会记录原因。
```
