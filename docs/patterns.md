# 设计模式与反模式

> 本文规范常见 Skill 架构模式、组合方式和反模式。

## 1. 三大原则

### 1.1 渐进披露

信息按需加载：

```text
L1  frontmatter
    name 和 description，供 Agent 判断是否触发。

L2  SKILL.md
    核心规则和执行地图，触发后读取。

L3  外部资源
    workflow/、references/、scripts/、assets/，按步骤读取。
```

### 1.2 单一职责

一个 Skill 围绕一类任务。复杂流程可以组合多个 Skill，但不要把一个 Skill 做成万能工具箱。

### 1.3 可移植性

公开 Skill 默认读者没有作者环境。因此：

- 不硬编码私有路径。
- 不硬编码私有账号。
- 不依赖未公开服务。
- 不把平台扩展写成通用标准。

## 2. 模式速查

```text
P1 顺序编排
  多步骤任务，Step 1 → Step 2 → Step 3。

P2 多工具协调
  多个 CLI、MCP 或 API 协同。

P3 迭代精化
  草稿 → 验证 → 修复 → 再验证。

P4 上下文感知分流
  按输入类型选择不同路径。

P5 领域知识封装
  把专业规则放入 references/。

P6 并行审查
  独立批次并行处理，再合并。

P7 可视化输出
  JSON 数据 → HTML 或 Markdown 报告。

P8 定时执行
  适合监控、巡检、周期性采集。

P9 链式组合
  Skill A 输出作为 Skill B 输入。

P10 公开发布审计
  安全、依赖、文档、测试统一门禁。
```

## 3. 典型模式

### 3.1 顺序编排

适合报告生成、发布前检查、文件处理、审查流程。

```text
workflow/
  step01-prepare.md
  step02-execute.md
  step03-verify.md
```

关键：

- 每步有输入输出。
- 每步更新状态。
- 失败可定位。

### 3.2 多工具协调

适合多来源采集、跨平台发布、多工具审计。

```text
step01-load-config
step02-call-tool-a
step03-call-tool-b
step04-merge-results
step05-report
```

关键：

- 每个工具输出中间文件。
- 不让工具之间直接互相依赖。
- 统一错误处理。

### 3.3 迭代精化

适合内容优化、代码修复、报告质量提升。

```text
draft
validate
fix
re-validate
finalize
```

关键：

- 有退出条件。
- 有最大迭代次数。
- 每轮产物有版本或最新指针。

### 3.4 上下文感知分流

适合多文件类型处理、多模式审查、多平台适配。

```text
输入为 Markdown → step02-markdown
输入为 PDF      → step02-pdf
输入为目录     → step02-directory
```

关键：

- 分流条件客观。
- 分流理由写入状态。
- 未命中时有兜底。

### 3.5 领域知识封装

适合法务审查、安全审计、平台发布规则、品牌格式规范。

```text
references/
  rules/
  examples/
  templates/
```

关键：

- 专业知识可读。
- 规则可验证。
- 公开版不含私有客户资料。

### 3.6 并行审查

适合多文件独立审查、多候选方案评估、多来源材料整理。

```text
step01-split/
step02-batch-a/
step02-batch-b/
step03-merge/
```

关键：

- 批次互不写同一文件。
- 合并步骤统一去重。
- 并行失败可单独重试。

### 3.7 可视化输出

适合报告、仪表盘、对比结果。

```text
data.json
template.html
report.html
```

关键：

- 数据和模板分离。
- HTML 不泄漏本地路径。
- 可回退为 Markdown 报告。

### 3.8 链式组合

适合端到端工作流。

规则：

- Skill A 输出稳定文件。
- Skill B 输入读取该文件。
- 不依赖聊天上下文传递数据。

## 4. 反模式

### 4.1 万能助手

一个 Skill 试图处理所有任务，导致触发混乱。

修复：

- 拆成多个 Skill。
- description 缩窄。
- 明确排除项。

### 4.2 入口臃肿

`SKILL.md` 放了全部教程、案例和排错。

修复：

- 保留入口契约。
- 长内容拆到 `docs/` 或 `references/`。

### 4.3 无状态长任务

长任务没有 `runs/` 和 `progress.json`。

修复：

- 创建运行目录。
- 每步写产物。
- 更新恢复提示。

### 4.4 脚本黑盒

脚本不可独立运行，失败也返回成功。

修复：

- 明确参数。
- 输出 JSON。
- 失败返回非零退出码。

### 4.5 私有依赖

公开示例依赖作者本机路径或账号。

修复：

- 换成占位路径。
- 改成环境变量。
- 写清用户需要提供什么。

### 4.6 没有测试

只靠人工感觉验证。

修复：

- 增触发正例和反例。
- 增主路径测试。
- 增安全扫描。

## 5. 检查清单

```text
选择
  [ ] 模式匹配任务复杂度。
  [ ] 没有为了完整而过度设计。
  [ ] 多模式有清晰路由。

组合
  [ ] 不同模式通过文件传递数据。
  [ ] 并行任务写入独立目录。
  [ ] 合并步骤有验证。

反模式
  [ ] 不是万能助手。
  [ ] 入口不过长。
  [ ] 长任务有状态。
  [ ] 公开版无私有依赖。
```
