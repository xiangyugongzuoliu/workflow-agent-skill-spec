# 质量评分规范

> 本文定义 Workflow Agent Skill 的百分制评分、等级裁决和阻塞项。

## 1. 评分目标

评分不是为了给文档打分，而是判断一个 Skill 是否值得被安装、复用、公开发布或放入团队标准。

评分应回答四个问题：

- Agent 能不能稳定触发。
- 陌生用户能不能运行。
- 中断后能不能恢复。
- 公开发布是否可审计。

## 2. 等级裁决

```text
A 90-100
  可公开推荐。结构完整，主路径可复现，安全和恢复都有验证。

B 75-89
  可公开发布。存在非阻塞问题，但不影响核心任务。

C 60-74
  仅适合内部试用。需要补测试、runtime、恢复或安全边界。

Blocked <60
  不应发布。触发、依赖、安全或可复现性存在阻塞问题。
```

存在阻塞项时，即使总分较高，也应裁决为 `blocked`。

## 3. 百分制维度

```text
入口契约 15 分
  name 稳定、description 清楚、范围和排除场景明确。

工作流结构 20 分
  workflow 步骤可执行，每步有输入、输出、完成标准和失败处理。

Runtime 与 setup 15 分
  环境、二进制、模型、cache、网络需求可声明、可诊断、可准备。

状态恢复 15 分
  runs/、state/config.json、progress.json、resume_hint 支持中断恢复。

安全边界 15 分
  无真实凭据、无私人路径、无客户数据、破坏性动作默认 dry-run。

测试评估 15 分
  触发、功能、恢复、安全、runtime 和跨模型评测有样例或执行记录。

文档体验 5 分
  README、用户指南、示例和发布清单一致，陌生用户能快速开始。
```

## 4. 阻塞项

以下任一项直接阻塞公开发布：

- description 无法判断何时触发。
- 示例依赖维护者本机路径、私有账号或未公开服务。
- 仓库包含真实凭据、客户数据或私人工作目录。
- 默认执行删除、上传、发布、付款、发信或修改远端系统。
- 需要 runtime、模型或系统二进制，但没有声明或诊断方式。
- 长任务没有恢复状态，失败后只能依赖聊天记录。
- README 承诺了没有测试过的能力。
- 脚本失败后仍返回成功。

## 5. 评分记录格式

建议在审查报告中使用：

```json
{
  "score": 86,
  "grade": "B",
  "decision": "ready-with-notes",
  "dimensions": {
    "entry_contract": 14,
    "workflow_structure": 18,
    "runtime_setup": 12,
    "run_state": 13,
    "security": 15,
    "testing": 10,
    "documentation": 4
  },
  "blockers": [],
  "notes": [
    "runtime doctor 有说明，但没有提供实际输出样例。"
  ]
}
```

## 6. 检查清单

```text
裁决
  [ ] 已给总分。
  [ ] 已给 A/B/C/Blocked 等级。
  [ ] 已单独列出阻塞项。

证据
  [ ] 每个扣分项有文件或行为证据。
  [ ] 未用“感觉不错”替代可验证断言。

发布
  [ ] blocked 不允许公开发布。
  [ ] B/C 必须列出发布前或发布后修复项。
```
