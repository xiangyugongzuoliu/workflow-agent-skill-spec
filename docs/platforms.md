# 跨平台兼容规范

> 本文规范 Workflow Agent Skill 如何处理不同 Agent 客户端、工具系统、frontmatter 字段、项目入口和运行能力差异。

## 1. 职责

平台规范解决：

- 哪些规则是通用原则。
- 哪些字段是平台扩展。
- 如何避免把单个平台能力当作通用标准。
- 如何写兼容多个 Agent 客户端的公开说明。

## 2. 通用规范层

多数 Workflow Agent Skill 都应遵守：

- 入口文件清晰。
- `description` 写清触发和排除。
- 长内容按需读取。
- `workflow/` 步骤有输入、输出、执行者和完成标准。
- 状态可恢复。
- 凭据不硬编码。
- 脚本和工具边界可审计。
- 触发、功能、恢复和安全可以测试。
- 公开示例可复现。

这些是通用工程原则，不依赖特定客户端。

## 3. 平台适配层

不同客户端可能支持：

- 额外 frontmatter 字段。
- 预批准工具。
- hooks。
- SubAgent 或隔离上下文。
- 插件打包。
- 上传包。
- 项目规则文件。
- 浏览器或系统工具。

公开规范应写成条件式能力：

```text
如果客户端支持 SubAgent 或隔离上下文，可用于隔离长材料分析。
如果不支持，则由主 Agent 分阶段读取并压缩上下文。
```

不要写成：

```text
所有 Skill 必须使用某个平台的 SubAgent 能力。
```

## 4. frontmatter 兼容

通用字段建议保持少而稳：

```text
name
description
license
compatibility
metadata
```

平台扩展字段应单独说明：

```text
allowed-tools
  某些客户端用于预批准工具、工具声明或能力提示，含义以客户端文档为准。

hooks
  某些客户端支持生命周期钩子。

model
  某些客户端支持模型覆盖。

effort
  某些客户端支持推理强度覆盖。
```

规则：

- 核心能力不依赖扩展字段。
- 扩展字段有降级方案。
- 文档写清适用平台。
- 未确认目标客户端支持时，不把扩展字段写进最小模板。

## 5. 工具兼容

工具分三类：

```text
本地命令
  find、rg、python、node、gitleaks 等。

客户端工具
  由 Agent 客户端提供的文件、终端、浏览器、SubAgent、MCP 工具。

外部服务
  GitHub、CMS、搜索、数据库、API。
```

公开 Skill 应声明：

- 必需工具。
- 可选工具。
- 缺失工具时的降级路径。
- 工具是否会读写文件或访问网络。

## 6. 客户端示例不是标准本身

公开规范可以兼容 Claude Code、Codex、Cursor、OpenCode、Gemini CLI 或其他 Agent 客户端，但不要把其中任何一个客户端的私有能力写成通用要求。

推荐写法：

```text
某些客户端可能支持额外 frontmatter 字段、hooks 或模型覆盖。
某些客户端可能通过 AGENTS.md、CLAUDE.md、项目规则目录或本地技能目录读取上下文。
以当前客户端实际支持为准。
```

禁止写法：

```text
所有客户端都会读取 CLAUDE.md。
所有客户端都会读取 AGENTS.md。
所有客户端都支持 hooks、allowed-tools 或 SubAgent。
```

## 7. 项目规则文件

常见可选入口：

```text
README.md
  面向人类读者。

AGENTS.md
  面向支持 AGENTS.md 的 Agent 客户端。

CLAUDE.md
  面向支持 CLAUDE.md 的 Agent 客户端。

其他规则文件或目录
  面向对应客户端，例如项目 rules、插件配置或本地技能目录。

SKILL.md
  Skill 入口。
```

职责边界：

- 项目长期规则放入对应客户端支持的项目上下文入口。
- 任务型能力放 `SKILL.md`。
- 读者说明放 `README.md`。

不要把项目规则全部塞进 Skill。

## 8. 平台差异记录

如果某个功能依赖平台能力，写清：

```text
功能：SubAgent 并行审查
依赖：客户端支持 SubAgent、并行任务或隔离上下文
降级：主 Agent 串行处理每个批次
风险：速度变慢，但结果结构不变
```

## 9. 公开分发

公开分发时：

- 不假设读者使用同一客户端。
- 不假设读者有同一工具。
- 不假设读者有同一模型。
- 不假设读者有作者账号。
- 不假设读者有作者目录结构。

把这些都变成显式依赖或可选能力。

## 10. 禁止事项

- 声称所有平台都支持同一字段。
- 没有降级路径就依赖 hooks。
- 没有说明就访问网络。
- 没有说明就调用外部 API。
- 没有说明就要求某个付费工具。
- 把项目规则当成 Skill 触发规则。

## 11. 检查清单

```text
通用
  [ ] 核心能力不依赖单一客户端扩展。
  [ ] 扩展字段标注适用平台。
  [ ] 必需工具和可选工具分开写。

降级
  [ ] SubAgent 或隔离上下文不可用时有串行方案。
  [ ] 外部工具不可用时有停止或降级说明。
  [ ] 网络不可用时不伪造结果。

公开
  [ ] 不要求作者本机环境。
  [ ] 不要求作者私有账号。
  [ ] README 没有过度承诺兼容性。
```
