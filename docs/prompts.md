# 提示词与 SubAgent 规范

> 本文规范 Skill 中提示词模板、SubAgent 指令、输出格式和失败处理的写法。

## 1. 职责

提示词用于定义 Agent 或 SubAgent 的判断任务。它不应承担确定性工作，也不应替代脚本、配置和状态文件。

适合提示词：

- 分析。
- 审查。
- 归纳。
- 生成建议。
- 判断严重程度。
- 选择执行路径。

不适合提示词：

- 批量文件扫描。
- JSON 解析。
- 链接检查。
- 重命名。
- 大量格式转换。

## 2. 放置位置

短提示词可以写在步骤文件中。长提示词放入：

```text
references/prompts/
  prompt-review.md
  prompt-summarize.md
```

步骤文件只引用路径：

```text
使用 references/prompts/prompt-review.md。
输入为 step02-scan/findings.json。
输出写入 step03-review/review.json。
```

## 3. 提示词基本结构

```markdown
# 任务

你要完成什么。

## 输入

读取哪些文件。

## 判断标准

按哪些规则判断。

## 输出格式

写到哪个文件，结构是什么。

## 禁止事项

不能做什么。

## 失败处理

缺输入、格式错误、证据不足时怎么办。
```

## 4. SubAgent 输出原则

SubAgent 输出应是下一步可直接使用的执行指令。

推荐：

```json
{
  "decision": "blocked",
  "blocking_issues": [
    {
      "file": "config/example.json",
      "evidence": "包含疑似真实 token 字段",
      "fix": "替换为环境变量占位符"
    }
  ],
  "next_actions": [
    "修改示例配置",
    "补 docs/setup.md 中的环境变量说明"
  ]
}
```

不推荐：

```text
这个项目整体不错，但还有一些地方可以提升。
```

## 5. 输出格式

能结构化就结构化。

适合 JSON：

- 审查结果。
- 分数。
- 问题列表。
- 批处理结果。
- 状态信息。

适合 Markdown：

- 面向人读的报告。
- 最终说明。
- 教程。
- 长文档。

如果两者都需要，先写 JSON，再由 Agent 或脚本渲染 Markdown。

## 6. 证据要求

审查类提示词必须要求证据：

- 文件路径。
- 行号或片段。
- 影响说明。
- 修复建议。

没有证据的问题应标为“待确认”，不能写成确定结论。

## 7. 失败处理

提示词必须定义失败行为：

```text
如果输入文件不存在：
  停止并返回 missing_input。

如果证据不足：
  标记 needs_manual_review。

如果发现疑似密钥：
  脱敏值，只报告键名和路径。
```

## 8. 迭代类提示词

需要多轮改进时，必须有退出条件。

```text
最多迭代 3 轮。
每轮必须写入 feedback_round_N.md。
通过全部检查后停止。
仍失败则输出 remaining_risk。
```

不要无限循环。

## 9. 主 Agent 与 SubAgent 分工

```text
主 Agent
  - 读取 SKILL.md。
  - 编排步骤。
  - 管理运行目录。
  - 合并结果。
  - 输出最终答复。

SubAgent
  - 读取指定材料。
  - 做局部判断。
  - 输出短结果。
  - 不修改主流程外文件。
```

SubAgent 不应拿到不必要的全局上下文。

## 10. 禁止事项

- 提示词没有输出格式。
- 提示词要求“尽量做好”但没有验收标准。
- 提示词让 SubAgent 读取无关目录。
- 提示词要求输出长篇背景分析。
- 提示词打印完整密钥。
- 提示词绕过用户授权执行破坏性动作。
- 提示词把私有知识写死。

## 11. 检查清单

```text
结构
  [ ] 有任务。
  [ ] 有输入。
  [ ] 有判断标准。
  [ ] 有输出格式。
  [ ] 有失败处理。

证据
  [ ] 审查结论有路径和证据。
  [ ] 证据不足会标记不确定。

分工
  [ ] 确定性任务不交给提示词。
  [ ] SubAgent 只读取必要材料。
  [ ] 输出可被下一步直接使用。
```
